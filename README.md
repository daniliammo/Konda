# Konda

Konda - экспериментальный русский язык программирования, который сейчас
транспилируется в C и затем собирается обычным C-компилятором.

Проект находится на раннем этапе. Уже есть лексер, парсер с построением
AST (рекурсивный спуск + Pratt для приоритетов операторов), отдельные
проходы семантики (проверка выходов за границы `аргументы[]` на этапе
транспиляции) и кодогена, а также минимальные автотесты.

## Пример

```конда
#содержит <stdio.h>
#содержит <stdlib.h>

целое32 точка_входа(целое32 количество_аргументов, символ** аргументы)
{
    если (количество_аргументов > 2) {
        целое32 левое = atoi(аргументы[1])
        целое32 правое = atoi(аргументы[2])

        printf("%d + %d = %d\n", левое, правое, левое + правое)
        printf("%d - %d = %d\n", левое, правое, левое - правое)
        printf("%d * %d = %d\n", левое, правое, левое * правое)

        если (правое > 0) {
            printf("%d / %d = %d\n", левое, правое, левое / правое)
        } иначе {
            printf("деление на ноль не выполняю\n")
        }
    } иначе {
        printf("использование: ./вывод/тест.elf 10 5\n")
    }
}
```

Транспилятор превращает `точка_входа` в `main`, а `#содержит` в
обычный `#include`.

## Сборка

Проект использует стандарт C23 (`-std=gnu23`) и его возможности
(`nullptr`, `[[maybe_unused]]`), поэтому нужен достаточно свежий
компилятор: **GCC ≥ 14** или **Clang ≥ 18**.

Проверить версию перед сборкой:

```sh
gcc --version   # или: cc --version
```

```sh
make
```

После сборки появятся:

```sh
Собранное/libkonda.so   # разделяемая библиотека: весь конвейер транспиляции
Собранное/Транспилятор  # CLI поверх библиотеки (линкуется с libkonda.so)
```

Логика транспиляции вынесена в библиотеку **libkonda**; CLI (`основа.c`) —
тонкая обёртка над её публичным API. Публичный заголовок — `konda.h`
(единственный, который подключает пользователь библиотеки):

```c
#include <konda.h>
// токенизировать_код → разобрать → анализировать → кодоген → скомпилировать_си
```

`make install` ставит `libkonda.so`, заголовки и CLI (`konda`) в `PREFIX`
(по умолчанию `/usr/local`).

## Запуск

```sh
./Собранное/Транспилятор tests/тест.конда
./вывод/тест.elf 10 5
```

Сгенерированный C и собранный бинарник складываются в папку `вывод/`
(она не коммитится), чтобы не засорять корень проекта.

### Несколько файлов

Транспилятор принимает любое число `.конда`-файлов в одной команде и
собирает их как одну программу — без заголовочных файлов и `extern`-объявлений:

```sh
./Собранное/Транспилятор главный.конда модуль.конда
./вывод/главный.elf
```

Имя выходного `.c`/бинарника берётся от первого аргумента. Семантика и
кодоген видят объединённое AST, поэтому функция из одного файла спокойно
вызывается из другого, а структура из соседнего файла видна в объявлениях
(парсер делает signature-pass по именам структур до полноценного разбора).
Кодоген эмитит forward-прототипы всех пользовательских функций до их тел,
так что порядок файлов в командной строке роли не играет.

Ожидаемый вывод:

```text
10 + 5 = 15
10 - 5 = 5
10 * 5 = 50
10 / 5 = 2
```

## Тесты

```sh
make test
```

Тесты проверяют:

- сборку транспилятора;
- что `тест.конда` (песочница) транспилируется и компилируется;
- передачу аргументов `printf` как есть, без подстановки `"%s\n"`;
- поддержку условия `>=`;
- приоритет операторов (`2 + 3 * 4 == 14`);
- вложенные `если` / `иначе`;
- объявления переменных и арифметику над ними;
- целочисленные типы по разрядности (`целое8/16/32/64` → `int8_t`…`int64_t`);
- остальные примитивы (`байт` → `uint8_t`, `вещественное` → `float`, `символ` → `char`);
- многоаргументные вызовы (`printf` с тремя и более аргументами);
- циклы `для`/`пока`, присваивание, унарный минус и `вернуть`;
- указатели в теле функции (`символ* …` → `char *…`);
- модификатор `неподписанный` (`целоеN` → `uintN_t`) и ошибку для нецелого типа;
- ошибки таблицы символов: необъявленная переменная, повторное объявление,
  выход переменной за область видимости, индексирование не-указателя;
- ключевое слово `нуль` → `nullptr` в сгенерированном C;
- нулевое разыменование указателя на этапе компиляции: ошибку при
  индексировании указателя, доказанно равного `нуль` (явный `= нуль` и
  неинициализированный), зануление сырого указателя в `nullptr` в кодогене,
  отсутствие ложных срабатываний после переприсваивания валидным значением
  и для ненулевого указателя;
- путь-чувствительный анализ нулевости и инициализации: ошибку для указателя,
  присвоенного валидным значением лишь в одной ветви `если` или только в теле
  цикла (после блока может быть нулём), компиляцию при присваивании в обеих
  ветвях, такую же проверку для definite-assignment скаляров, и корректную
  работу sink-ветви (безусловный `вернуть` не вливается в слияние);
- инициализацию до использования: ошибку при чтении неинициализированного
  скаляра (напрямую и в выражении), отсутствие ложных срабатываний после
  присваивания и для занулённого массива;
- проверку диапазона константы-инициализатора: ошибки для константы и
  свёрнутого выражения вне диапазона типа, отрицательной константы в
  беззнаковый тип, и компиляцию граничных значений (255, -128, 65);
- ошибку для небезопасного доступа к `аргументы[1]`;
- инициализацию массива списком значений (полный/частичный литерал,
  выражения как элементы, ошибки переполнения и несовместимого типа,
  сохранение поведения зануления без инициализатора);
- авто-конверсию массива в срез при передаче в функцию: построение
  дескриптора без аллокаций, ошибку для несовместимого типа элемента,
  сохранение прежнего поведения для указательного и срез-аргумента;
- multi-file: вызов функции из соседнего файла (forward-прототип в C),
  структура из одного файла, используемая в объявлении переменной в другом
  (signature-pass по именам структур), независимость от порядка аргументов
  в командной строке;
- `для_каждого` по срезу и массиву: обход всех элементов, использование
  элемента в выражении (сумма), отсутствие рантайм-проверки границ в теле
  цикла, ошибка для не-коллекции, запрет move-из-внешней-области внутри
  тела;
- path-sensitive move владения: симметричный move в обеих ветвях
  `если`/`иначе`, ошибку для асимметричного move, sink-`вернуть` в ветви
  как корректный паттерн, запрет move из объемлющей области в теле цикла
  (для `пока` и `для`), разрешение move среза, объявленного внутри тела
  цикла, диагностику use-after-move с указанием строки места move;
- функцию-потребитель (`владеет`-параметр): запрет передачи владельца из
  объемлющей области в потребитель внутри тела цикла (`для` и `пока`) как
  use-after-move, и разрешение для среза, объявленного внутри тела цикла;
- транзитивную деструкцию структур: вызов деструктора поля-структуры при
  выходе из области (даже без собственного `[удалить]` у объемлющей
  структуры), обратный порядок освобождения нескольких полей и запрет
  копирования структуры, транзитивно содержащей деструктор;
- перечисления (enum): базовое объявление и сравнение (авто-нумерация), явные
  значения с продолжением авто-инкремента, перечисление как параметр функции и
  поле структуры, `enum → целое` (расширение), ошибки для `целое → enum`,
  смешения разных перечислений и чтения перечисления до инициализации;
- корректную ошибку для несуществующего файла.

## Что уже поддерживается

- директива `#содержит <...>`;
- функция `точка_входа`;
- типы `целое8`, `целое16`, `целое32`, `целое64`, `байт`, `вещественное`, `символ`;
- модификатор `неподписанный` для целых типов (`неподписанный целое32` → `uint32_t`);
- завершение операторов переводом строки — `;` в конце строки не нужна
  (модель как в Go/Swift): перевод строки завершает оператор, если строка
  синтаксически закончена; строка, кончающаяся бинарным оператором или
  запятой, и любые переносы внутри `(…)`/`[…]` считаются продолжением.
  `;` остаётся как разделитель нескольких операторов в одной строке и как
  разделитель частей заголовка `для(…; …; …)`;
- `если` / `иначе`;
- циклы `пока` (→ `while`) и `для` (→ `for`);
- foreach: `для_каждого (имя из коллекция) { … }` — обход среза или массива
  фикс. размера; имя — фрешая переменная типа элемента (копия по значению,
  модификация не отражается на коллекции). Кодоген использует собственный
  индекс цикла, доступ прямой — без рантайм-проверок границ (нулевая
  стоимость, как у канонического `для` с BCE). Move-из-внешней-области
  внутри тела `для_каждого` запрещён по тем же причинам, что в обычном
  цикле;
- `вернуть` (→ `return`) — с выражением или без;
- `статический` (→ `static`): для функций — внутренняя связь
  (`статический целое32 f()` → `static int32_t f()`); для локальных
  переменных — статическая длительность, значение сохраняется между вызовами
  (`статический целое32 счётчик = 0`). Инициализатор статической переменной
  должен быть константой; `точка_входа` и срезы статическими быть не могут;
- объявления простых переменных и присваивание существующим;
- указатели в объявлениях тела функции (`тип* имя`, `тип** имя`);
- вызовы функций;
- вызовы `printf(...)` (без особой обработки: формат-строку задаёт сам код);
- выражения с идентификаторами, числами, строками, унарным минусом и простыми операторами;
- доступ к массиву с числовым литералом, например `аргументы[1]`;
- таблица символов с областями видимости: проверка обращения к необъявленным
  переменным, повторного объявления в одной области и видимости по блокам;
- проверка совместимости типов по правилам C (без неявных приведений):
  ошибка при `указатель ← целое`, `целое ← указатель`, несовместимых типах
  указателей, несовместимом типе возврата, недопустимых операндах бинарного
  оператора и унарного `-`; разрешены `нуль → указатель`, смешение целых
  типов, арифметика указателей (`ptr ± целое`, `ptr − ptr`);
- защита сырых указателей на этапе компиляции (путь-чувствительно): нельзя
  индексировать указатель, который может быть `нуль` хотя бы на одном пути
  исполнения. Указатель нулевой, если объявлен `= нуль`/без инициализатора или
  переприсвоен `нуль`; присваивание валидного значения снимает это. На стыке
  ветвей состояние сливается честно: после `если` указатель «не нулевой»,
  только если он присвоен валидным значением в **обеих** ветвях; если хотя бы
  в одной он остаётся нулём (или присвоен лишь в теле цикла, который мог не
  выполниться) — разыменование после блока ошибка. Ветвь, заканчивающаяся
  безусловным `вернуть` (sink), в слияние не вливается. Дополнительно кодоген
  зануляет сырой указатель без инициализатора в `nullptr`;
- инициализация до использования (definite assignment, путь-чувствительно):
  скалярную переменную нельзя читать, пока она не присвоена на **всех** путях.
  Присваивание лишь в одной ветви `если` (или только в теле цикла) не делает
  её инициализированной — чтение после блока ошибка «используется до
  инициализации». Закрывает чтение неопределённой памяти (UB) статически, без
  рантайма. Массивы/структуры зануляются и считаются инициализированными;
- проверка диапазона константы-инициализатора: если сворачиваемая в константу
  величина не помещается в объявленный целый тип (`целое8 х = 1000`,
  `неподписанный целое8 = -1`, `целое16 = 300 * 200`), — ошибка «вне диапазона
  типа», вместо тихого усечения в C;
- пользовательские структуры: `структура Имя { тип поле … }` → `typedef
  struct`; объявление `Имя п` (зануляется), доступ к полям `п.поле` (чтение и
  запись; для указателя на структуру `.` транслируется в `->`), вложенные
  структуры (`о.начало.х`), передача структуры в функцию по значению; проверка
  существования структуры, полей и типов полей;
- перечисления (enum): `перечисление Имя { ВАРИАНТ [= значение] … }` →
  `typedef enum { … } Имя;`. Варианты — по одному на строку (модель A), с
  необязательным явным целым значением (иначе предыдущее + 1, как в C).
  Переменная `Имя ц = ВАРИАНТ`, сравнение `ц == ВАРИАНТ`, перечисление как
  параметр функции и поле структуры. Типобезопасность: присваивание/сравнение
  только в пределах одного перечисления, `enum → целое` разрешено (расширение),
  `целое → enum` и смешение разных перечислений — ошибка; вариант — это
  целочисленная константа (сворачивается, годится для значений и сравнений).
  Переменную-перечисление, как любой скаляр, нужно инициализировать до чтения;
- литералы структур (designated initializers C): `Имя п = { поле = значение,
  … }` — порядок полей не важен, незаданные поля зануляются автоматически
  (как `{ .поле = знач, … }` в C). Семантика проверяет существование каждого
  поля и совместимость типов значений. Инициализировать литералом можно
  только саму структуру, не указатель на неё;
- деструкторы структур (RAII, в стиле Vala): атрибут `[удалить = функция]`
  перед `структура` задаёт деструктор, который компилятор вызывает
  автоматически (`удалить(&п)`) при выходе переменной из области и перед
  `вернуть`, в обратном порядке объявления — без рантайма. Структуры без
  деструктора не порождают вызовов. Деструкция **транзитивна**: если поле
  структуры само является структурой с деструктором (или с таким полем),
  деструктор поля вызывается автоматически (`удалить(&п.поле)`), даже если у
  объемлющей структуры собственного `[удалить]` нет; поля освобождаются в
  обратном порядке объявления. Структуру, которой нужна деструкция
  (собственная или транзитивно через поле), нельзя копировать/присваивать или
  передавать по значению — это привело бы к двойному вызову деструктора;
- срезы `срез<T>` — толстый указатель `{данные, длина}` с защитой от
  переполнения буфера: `выделить(N)` создаёт срез, `длина(с)` даёт длину,
  доступ `с[i]` проверяется. Константный выход за границу ловится на этапе
  компиляции (как `аргументы[N]`), динамический — в рантайме (одно сравнение
  на доступ, как в Go/Rust; при нарушении — сообщение и `abort`). Срезы
  передаются в функции по типу `срез<T>`;
- autofree срезов (RAII на этапе компиляции, без рантайма): владеющий срез
  (из `выделить`) освобождается автоматически (`free`) в конце своей области
  видимости и перед каждым `вернуть`, в обратном порядке объявления — утечек
  нет. Параметр-срез заимствуется и в функции не освобождается (владелец
  снаружи);
- перемещение владения срезом (move) — без рантайма и без аннотаций:
  `срез<T> б = а` переносит владение в `б` (копируется только дескриптор,
  данные не трогаются — бесплатно), источник `а` становится недействительным;
  использование после перемещения — ошибка компиляции (use-after-move) с
  указанием строки места move. Длина переносится вместе с владением, поэтому
  статическая проверка границ и элиминация работают и после move.
  Освобождается ровно один владелец. Передача в функцию остаётся
  заимствованием (срез после вызова жив). Перемещение заимствованного
  среза (параметра) запрещено;
- path-sensitive move в `если`/`иначе`: владение может уйти в обеих ветвях,
  тогда после if срез считается перемещённым (autofree не вставляет
  освобождение). Если move есть только в одной ветви, итог двусмыслен —
  ошибка компиляции с указанием конфликтующей ветви. Ветвь, заканчивающаяся
  безусловным `вернуть`, в слияние не участвует (sink) — паттерн
  «потребить + ранний return в одной ветви» корректен. Без рантайм-флагов:
  компилятор либо доказал согласованность, либо отказал;
- move источника из объемлющей области внутри тела цикла (`пока`/`для`)
  запрещён: на втором витке источник уже был бы перемещён → use-after-move.
  Запрет действует одинаково и для move-через-присваивание (`срез б = а`), и
  для move-через-передачу в `владеет`-параметр (`поглотить(а)`). Move среза,
  объявленного *внутри* тела цикла, по-прежнему разрешён — новый владелец
  создаётся на каждом витке;
- возврат среза из функции — передача владения вызывающему: функция с типом
  возврата `срез<T>` отдаёт владение наружу (внутри не освобождается), а
  принимающий `срез<T> д = создать(...)` получает владение и освобождает срез
  один раз. Вернуть заимствованный (параметр) срез нельзя — функция им не
  владеет;
- переприсваивание срезу: `а = выделить(N)` (переаллокация), `а = клон(б)`,
  `а = в` (move), `а = функция()` — прежние данные цели освобождаются, цель
  получает новое владение; ни утечки, ни double-free;
- `клон(а)` — явная глубокая копия среза: новый независимый владелец с
  собственными данными (источник остаётся жив). Нужен, только когда правда
  требуются две копии данных — move и заимствование остаются бесплатными;
- функция-потребитель: параметр `владеет срез<T>` забирает владение —
  функция освобождает срез сама, а передача в такой параметр перемещает
  аргумент (после вызова он недействителен). Обычный параметр-срез по-прежнему
  заимствует (срез после вызова жив);
- массивы фиксированного размера `тип имя[N]` (как `[T; N]` в Rust): размер
  известен на этапе компиляции, массив зануляется по умолчанию. Литеральный
  (или сворачиваемый в константу) индекс проверяется на этапе компиляции —
  выход за границу не скомпилируется; динамический индекс проверяется в
  рантайме (длина берётся через `sizeof`, в структуру не пишется — нулевая
  цена по памяти). При литеральных индексах рантайм-проверка не генерируется
  вовсе;
- инициализация массива списком значений: `тип имя[N] = { выр, выр, … }` →
  агрегатный инициализатор C `{ … }`. Если значений меньше размера,
  хвостовые элементы C-компилятор зануляет автоматически (без собственного
  рантайма). Семантика проверяет, что элементов не больше `N` и тип каждого
  совместим с типом массива;
- авто-конверсия массива в срез при передаче в функцию: если параметр —
  `срез<T>`, а аргумент — переменная-массив `T[N]`, на месте вызова кодоген
  строит дескриптор `(Срез){ .данные = а, .длина = N, .эл_размер = sizeof(T) }`.
  Массив остаётся на стеке — никаких аллокаций; одна функция-«вью»
  (`сумма`, `найти`, `печатать`) работает и с массивами, и со срезами.
  Семантика проверяет совпадение типа элемента — иначе ошибка компиляции;
- устранение лишних проверок границ (bounds-check elimination): если
  компилятор доказал, что индекс заведомо в границах, рантайм-проверка не
  генерируется. Покрыто: константный индекс в границах известной длины; и
  канонический цикл `для (и = 0; и < длина(с); и = и + ш)` либо
  `для (и = 0; и < N; и = и + ш)` (ш > 0), в теле которого индекс и срез не
  переназначаются, — все доступы `с[и]` внутри безопасны. В неоднозначных
  случаях проверка сохраняется (консервативно), поэтому защита не теряется;
- защита от деления на ноль на этапе компиляции: свёртка констант ловит
  `x / 0`, `x % 0`, делитель-переменную, константно равную нулю
  (`зк = 0; … / зк`), и нулевые константные выражения (`… / (а - б)` при
  `а == б`); без рантайм-проверок — нулевая стоимость, как в Rust. Значение
  отслеживается только у неизменяемых переменных (которым не присваивают
  заново), поэтому ложных срабатываний нет;
- статическая проверка `аргументы[N]` внутри условий вида
  `количество_аргументов > N` и `количество_аргументов >= N`;
- диагностика ошибок с указанием строки и колонки (колонка считается
  по символам UTF-8, а не по байтам), а также ожидаемого и встреченного токена.

## Что пока не поддерживается

- строгая проверка сужения для переменных-выражений: целые смешиваются
  свободно, как в C (`целое32 в = целое8 + целое64`), запрещены лишь
  указатель↔целое и разнотипные указатели. Сужение ловится только для
  сворачиваемых в константу инициализаторов (см. «проверку диапазона»);
  запрет неявного сужения переменных потребовал бы синтаксиса явного
  приведения и строгого режима;
- защита формата `printf` от неконстантной строки (format-string): первый
  аргумент сейчас передаётся как есть по дизайну (`printf(аргументы[1])`);
  ограничение его строковым литералом — отдельный строгий режим;
- пользовательские функции с вызовом по кириллическому имени (транслитерация);
- перемещение среза из объемлющей области (`move` пока только в пределах
  одной области);
- массивы переменной длины и многомерные; вывод размера из литерала
  (`целое32 а[] = {1,2,3}`);
- автоматическая конвертация массива в срез действует только для параметра
  функции; в локальной переменной (`срез<T> с = а`) пока не поддерживается;
- слияние владения при сложном control-flow за пределами «sink-вернуть в
  ветви»: вложенный `если`, оба внутренних блока которого возвращают,
  считается sink только если последний оператор внешней ветви — сам
  `вернуть`; более глубокий анализ путей пока не делается;
- автоматическая конвертация массива в срез при передаче в функцию;
- срез как поле структуры (владеющий ресурс внутри структуры пока не
  отслеживается; копирование/перемещение структур с деструктором запрещено
  во избежание двойного вызова — см. раздел про транзитивную деструкцию);
- возврат пользовательской структуры или перечисления из функции по значению
  (тип в позиции возврата пока не разбирается);
- перечисление, объявленное в другом файле, как тип в соседнем `.конда`
  (имена перечислений пока не собираются между файлами, в отличие от структур);
- стабильная генерация ASCII-имен для всех кириллических идентификаторов.

## Архитектура

Транспиляция идёт конвейером из трёх проходов:

```
.конда (один или несколько) → лексер → разбор.c (AST на файл)
   → слияние AST → семантика.c (границы, типы) → кодоген.c → .c → cc
```

При нескольких входных файлах разбор делается независимо для каждого, потом
top-level дети сливаются в один общий `УЗЕЛ_ПРОГРАММА`. Перед разбором
делается пред-сканирование токенов всех файлов на «структура Имя», чтобы
парсер каждого файла знал о пользовательских типах из соседних. Семантика
и кодоген дальше работают с одним общим AST — никаких заголовков и
extern-деклараций между .конда не нужно.

Структура узла AST — в `аст.h`/`аст.c` поверх универсального `ДинМассив`.

Path-sensitive анализ владения (snapshot/merge для срезов в `если`/`иначе`,
sink-детектор для `вернуть`) вынесен в отдельный модуль `владение.c` с
внутренним заголовком `владение.h`. Семантика держит `Анализатор`/`Символ`
приватными и предоставляет модулю владения 5 узких аксессоров — изменение
анализа путей не требует править остальные проходы.

Весь конвейер собран в разделяемую библиотеку **libkonda** (`Собранное/libkonda.so`)
с единым публичным заголовком `konda.h`. `основа.c` — это CLI поверх библиотеки;
любая программа может подключить `konda.h` и вызвать конвейер напрямую.

## Ближайший план

1. Расширять тесты перед каждым изменением синтаксиса.
2. Автоосвобождение срезов (`autofree`) при выходе из области видимости.
3. Перемещение и заимствование владения: защита от use-after-free и double-free.

Целевая архитектура описана в `АРХИТЕКТУРА_ТРАНСПИЛЯТОРА.md`.
