# Konda

Konda - экспериментальный русский язык программирования, который сейчас
транспилируется в C и затем собирается обычным C-компилятором.

Проект находится на раннем этапе. Уже есть лексер, парсер с построением
AST (рекурсивный спуск + Pratt для приоритетов операторов), отдельные
проходы семантики (проверка выходов за границы `аргументы[]` на этапе
транспиляции) и кодогена, а также минимальные автотесты.

## Пример

```конда
#содержит <stdio.h>
#содержит <stdlib.h>

целое32 точка_входа(целое32 количество_аргументов, символ** аргументы)
{
    если (количество_аргументов > 2) {
        целое32 левое = atoi(аргументы[1])
        целое32 правое = atoi(аргументы[2])

        printf("%d + %d = %d\n", левое, правое, левое + правое)
        printf("%d - %d = %d\n", левое, правое, левое - правое)
        printf("%d * %d = %d\n", левое, правое, левое * правое)

        если (правое > 0) {
            printf("%d / %d = %d\n", левое, правое, левое / правое)
        } иначе {
            printf("деление на ноль не выполняю\n")
        }
    } иначе {
        printf("использование: ./вывод/тест.elf 10 5\n")
    }
}
```

Транспилятор превращает `точка_входа` в `main`, а `#содержит` в
обычный `#include`.

## Сборка

Проект использует стандарт C23 (`-std=gnu23`) и его возможности
(`nullptr`, `[[maybe_unused]]`), поэтому нужен достаточно свежий
компилятор: **GCC ≥ 14** или **Clang ≥ 18**.

Проверить версию перед сборкой:

```sh
gcc --version   # или: cc --version
```

```sh
make
```

После сборки появятся:

```sh
Собранное/libkonda.so   # разделяемая библиотека: весь конвейер транспиляции
Собранное/Транспилятор  # CLI поверх библиотеки (линкуется с libkonda.so)
```

Логика транспиляции вынесена в библиотеку **libkonda**; CLI (`основа.c`) —
тонкая обёртка над её публичным API. Публичный заголовок — `konda.h`
(единственный, который подключает пользователь библиотеки):

```c
#include <konda.h>
// токенизировать_код → разобрать → анализировать → кодоген → скомпилировать_си
```

`make install` ставит `libkonda.so`, заголовки и CLI (`konda`) в `PREFIX`
(по умолчанию `/usr/local`).

## Запуск

```sh
./Собранное/Транспилятор tests/тест.конда
./вывод/тест.elf 10 5
```

Сгенерированный C и собранный бинарник складываются в папку `вывод/`
(она не коммитится), чтобы не засорять корень проекта.

### Сборка: отладка и релиз

По умолчанию — **отладочная** сборка: включены все рантайм-проверки (границы
срезов/массивов/`аргументы`, guard'ы деления/сдвига/переполнения), компиляция
`cc -O0 -g`. Флаг `--релиз` (синоним `--release`) даёт **релизную** сборку:
компиляция `cc -O2`, а из рантайм-проверок снимаются только **арифметические
guard'ы** (переполнение `+`/`-`/`*`, деление/сдвиг) — **проверки границ
срезов/массивов/`аргументы` остаются** (безопасность памяти важнее скорости).
Статический анализ (типы, сужение, границы констант, владение, исчерпываемость
`выбор`) работает в обоих режимах одинаково — он бесплатен.

```sh
./Собранное/Транспилятор --релиз главный.конда   # -O2, без арифм. guard, границы на месте
./Собранное/Транспилятор главный.конда           # отладка, все проверки (по умолч.)
```

Модель как в Rust: проверки границ живут и в релизе (выход за границы → `abort`,
а не порча памяти), а debug-only остаётся дорогой контроль целочисленного
переполнения. В тесте `сумма += и` на 500 млн итераций релиз ≈ 0.12 с против
1.1 с в отладке (арифметический guard блокирует автовекторизацию; сняв его в
релизе, получаем скорость уровня C, не жертвуя безопасностью памяти).

### Сборка библиотеки

Флаг `--библиотека` (синонимы `--lib`, `--library`) собирает **разделяемую
библиотеку** `.so` вместо исполняемого файла — точка входа `точка_входа` при
этом не нужна. Рядом кладётся **публичный заголовок** `.h` (`#pragma once`,
типы-перечисления и структуры, тип `Срез` при необходимости и прототипы всех
НЕ-`статический` функций), так что библиотеку можно подключать из C:

```sh
./Собранное/Транспилятор --библиотека матем.конда         # → вывод/матем.so + вывод/матем.h
./Собранное/Транспилятор --библиотека --релиз матем.конда # то же, но -O2 без арифм. guard
```

```c
#include "вывод/матем.h"
int main(void){ return сложить(2, 3); }
/* cc main.c -L вывод -l:матем.so -Wl,-rpath,вывод -o main */
```

`статический`-функции остаются внутренними (в заголовок не попадают). Режимы
отладка/релиз действуют так же, как для исполняемого файла.

### Безопасное использование библиотеки из другого проекта Konda

При `--библиотека` рядом с `.so`/`.h` эмитится ещё и **Konda-интерфейс** `.и` с
полными Konda-сигнатурами (типы + владение: `владеет`/`изменяемый`/`вывод`,
элемент среза). Проект-потребитель, подключающий `#содержит "матем.h"`,
**автоматически** загружает соседний `матем.и` — и вызовы к функциям библиотеки
проверяются **как внутрипроектные** (типы и владение), а не как непрозрачный
внешний C. Транспилятор при этом сам **линкует** `матем.so` (с `-rpath`), так что
всё работает end-to-end:

```sh
./Собранное/Транспилятор --библиотека матем.конда   # → вывод/матем.{so,h,и}
./Собранное/Транспилятор программа.конда             # #содержит "вывод/матем.h"
./вывод/программа.elf                                 # линкована и запускается
```

Без `.и` (например, обычная C-библиотека) поведение прежнее — непрозрачный
внешний C (только имя+арность). Это разница между «путём 1» (подключить исходники
`.конда` — всё проверяется вместе) и безопасным «путём 2» (готовая `.so` +
интерфейс).

**Что даёт:** передача не-lvalue в `изменяемый`/`вывод`-параметр библиотеки,
неверный тип аргумента, нарушение `владеет` (move) — ловятся на уровне Konda с
позицией в `.конда`, а не молча уходят в C-компилятор.

**Ограничения v1:** покрываются примитивы, `срез<примитив>` и указатели. Экспорт с
типами `структура`/`перечисление`/`типфункции`, а также имена из пространств имён
и обобщения остаются непрозрачными (fallback к `.h`).

### Несколько файлов

Транспилятор принимает любое число `.конда`-файлов в одной команде и
собирает их как одну программу — без заголовочных файлов и `extern`-объявлений:

```sh
./Собранное/Транспилятор главный.конда модуль.конда
./вывод/главный.elf
```

Имя выходного `.c`/бинарника берётся от первого аргумента. Семантика и
кодоген видят объединённое AST, поэтому функция из одного файла спокойно
вызывается из другого, а структура **или перечисление** из соседнего файла
видны в объявлениях (парсер делает signature-pass по именам структур и
перечислений до полноценного разбора — симметрично для обоих). Кодоген
эмитит forward-прототипы всех пользовательских функций до их тел, так что
порядок файлов в командной строке роли не играет.

Ожидаемый вывод:

```text
10 + 5 = 15
10 - 5 = 5
10 * 5 = 50
10 / 5 = 2
```

## Безопасность: сырые указатели и `небезопасно { }`

Небезопасные операции с сырыми указателями запрещены в обычном коде **статически**
(на этапе компиляции, без рантайм-цены) и разрешены только внутри блока
`небезопасно { … }`. Уже запрещено везде: взятие адреса `&x`, фабрикация
указателя из целого (`как<T*>(целое)`), разыменование `*p`. Дополнительно теперь
гейтятся:

- **арифметика указателей** (`p + n`, `p - p`) — уводит указатель за пределы
  объекта без проверки границ;
- **индексация сырого указателя** (`p[i]`) — доступ к памяти без проверки границ;
- **реинтерпретация указателя** (`как<T*>(указатель_другого_типа)`) — type
  confusion: биты одного типа читаются как другой (каст указателя того же типа,
  например смены знаковости, разрешён);
- **небезопасные функции libc** (`strcpy`, `strcat`, `sprintf`, `gets`,
  `stpcpy`, `vsprintf`, `wcscpy`, `wcscat`) — неограниченная запись в буфер;
  безопасные (`printf`, `strlen`, `malloc`, …) не затронуты;
- **доступ к члену `союз`а** (`з.поле`) — активный член не отслеживается, чтение
  неактивного = type confusion. Безопасная альтернатива — `перечисление` +
  `выбор` (тег отслеживается).

```konda
целое32 точка_входа(целое32 количество_аргументов, символ** аргументы)
{
    символ* с = "abcd"
    // символ* д = с + 1     // ← ошибка компиляции в безопасном коде
    небезопасно {
        символ* д = с + 1    // арифметика — только здесь
        printf("%c\n", д[0]) // индексация сырого указателя — только здесь
    }
    вернуть 0
}
```

Что **не** гейтится (остаётся безопасным): индексация `срез<T>` и массивов
фиксированного размера (проверка границ есть), индексация `аргументы` (argv —
проверка против `количество_аргументов`), сравнение указателей. Идиоматический
путь — использовать `срез<T>` вместо сырого указателя; `небезопасно` нужен лишь
для низкоуровневого interop.

`небезопасно { }` — это в C обычный блок `{ }`: **никакой рантайм-цены**, гейт
чисто статический, поэтому в релизе всё работает так же быстро, как в C.

## Ненулевые указатели по умолчанию и `возможно<T*>`

Обычный указатель `T*` **ненулевой по умолчанию**: ему нельзя присвоить `нуль`
(ни при объявлении, ни присваиванием) и нельзя объявить без инициализатора. Его
можно разыменовывать/индексировать без проверки на `нуль` — он гарантированно не
нулевой.

Nullable-указатель объявляется как **`возможно<T*>`**: он может быть `нуль`.
Разыменовать его напрямую нельзя — сначала «разверни» под охранником:

```konda
возможно<символ*> п = нуль
если (количество_аргументов > 1) { п = аргументы[1] }

если (п != нуль) {
    небезопасно { printf("%c\n", п[0]) }   // здесь п доказанно ненулевой
} иначе {
    printf("нет значения\n")
}
```

Работает и симметричная форма `если (п == нуль) { … } иначе { /* здесь ненулевой */ }`.
Присвоить nullable-указатель обычному без сужения нельзя. Проверки **статические**
(путь-чувствительный анализ нулевости), нулевая рантайм-цена: `возможно<T*>` в C —
обычный указатель. Заодно добавлен оператор `!=` (сравнение «не равно»).

Опция-над-значениями (`возможно<целое>` и т.п.) — отдельная возможность (нужен
флаг присутствия), пока не реализована; `возможно` применимо к указателям.

## Иммутабельность: `конст`

Переменную можно объявить неизменяемой префиксом `конст`. Такую переменную нельзя
переприсвоить, изменить её элемент/поле или передать в параметр `изменяемый`/
`вывод`. `конст`-переменная обязана быть инициализирована при объявлении.
Проверка **статическая** (нулевая рантайм-цена — в C это обычная переменная).

```konda
конст целое32 предел = 100
предел = 200          // ← ошибка компиляции: «конст» неизменяема
```

Значение `конст`-переменной, инициализированной константой, участвует в свёртке
констант — это даёт больше статических доказательств границ (см. ниже).

## Знаковое переполнение в релизе (`-fwrapv`)

В отладке знаковое `+`/`-`/`*` с неконстантными операндами проверяется рантайм-
guard'ом (abort с сообщением). В **релизе** guard снимается ради скорости, но код
компилируется с `-fwrapv`: переполнение **определено** (обёртка по модулю 2ⁿ), а
не UB. Нулевая рантайм-цена на two's-complement машинах; убирает целый класс
неопределённого поведения.

## Устранение проверки границ по охраннику

Проверка границ `срез<T>`/массива остаётся и в релизе (безопасность памяти, как в
Rust), но снимается там, где индекс **доказанно** в границах: константа в
диапазоне, индекс канонического цикла, и — теперь — **беззнаковый** индекс под
охранником `если (и < длина(с))` (тогда `и ∈ [0, длина(с))`: `и ≥ 0` из
беззнаковости, `и < длина` из условия). Чем больше доказано статически, тем
меньше проверок в рантайме — безопасность та же, а код быстрее.

```konda
неподписанный целое32 и = …
если (и < длина(д)) {
    д[и] = 1        // прямой доступ, без рантайм-проверки границ
}
```

## Безопасные макросы: условная компиляция

Konda поддерживает **условную компиляцию** — выбор веток кода на этапе
компиляции. Это безопасно, потому что это _выбор кода_, а не текстовая
подстановка: включённый код полностью проходит типизацию и все проверки Konda.
**Функциональных макросов нет** (они небезопасны — гигиена, двойное вычисление);
для именованных констант в коде используйте `конст`.

Директивы (C-имена и русские синонимы), каждая — отдельной строкой:

| C            | Русский            | смысл                                  |
|--------------|--------------------|----------------------------------------|
| `#define ИМЯ [N]` | `#определить ИМЯ [N]` | задать флаг/константу (для `#if`)   |
| `#if ВЫР`    | `#если ВЫР`        | ветка, если выражение истинно           |
| `#ifdef ИМЯ` | `#опр ИМЯ`         | если макрос определён                   |
| `#ifndef ИМЯ`| `#не_опр ИМЯ`      | если макрос НЕ определён                |
| `#elif ВЫР`  | `#иначе_если ВЫР`  | иначе-если                              |
| `#else`      | `#иначе`           | иначе                                   |
| `#endif`     | `#конец`           | конец условия                           |

```konda
#содержит <stdio.h>
#определить ВЕРСИЯ 3
целое32 точка_входа(целое32 количество_аргументов, символ** аргументы)
{
#если ВЕРСИЯ >= 3 && !определён(СТАРОЕ)
    printf("новая\n")
#иначе
    printf("старая\n")
#конец
    вернуть 0
}
```

В выражении `#if`: целые литералы, имена макросов (значение или 0 если не
определён), `defined(ИМЯ)`/`определён(ИМЯ)`, операторы `! - * / % + - < > <= >=
== != && ||` и скобки. Флаги задаются и из командной строки: `-D ИМЯ` (значение 1)
или `-D ИМЯ=значение`. `#содержит` и прочие `#`-директивы проходят через
препроцессор как есть. Неактивные ветки **не попадают в C** — нулевая рантайм-цена.

## Конвертер C → Konda (`СиВКонда`)

`СиВКонда` — **отдельный самостоятельный** инструмент, который переводит
подмножество C в исходник Konda. Это обратное направление к транспилятору.

Главный принцип — **строгое разделение**: в бинарнике транспилятора
(`Транспилятор`) только транспиляция, в бинарнике конвертера (`СиВКонда`) только
конвертация. Конвертер **не линкует** `libkonda` и **не содержит** ни лексера,
ни семантики, ни кодогена Konda — он делает лишь синтаксический перевод. Все
проверки безопасности/типов выполняет транспилятор Konda **отдельной командой**
над сгенерированным `.конда`:

```sh
make
./Собранное/СиВКонда  примерc.c              # → вывод/примерc.конда  (только перевод)
./Собранное/Транспилятор вывод/примерc.конда # все проверки Konda + сборка .c/.elf
```

Конвертер сообщает только об ошибках синтаксиса C и о неподдержанных
конструкциях (с позицией в `.c`). Если перевод корректен, но результат
небезопасен по правилам Konda (например, функция не возвращает значение по всем
путям, выход за границы, использование до инициализации) — это обнаружит и
покажет транспилятор на втором шаге, а не конвертер.

### Что переводится

Функции, глобальные и локальные переменные, `struct`/`union`/`enum`,
`typedef` (в т.ч. `typedef struct {…} N;` и указатель на функцию
`typedef R (*Op)(A,B);` → `типфункции`), `if/else`, `while`, `for`, `do-while`,
`switch`, `return/break/continue`, выражения (арифметика, сравнения, `&&`/`||`,
индексация, `.`/`->`, `sizeof`, приведения, тернарник), строковые/числовые/
символьные литералы. Автоматически: `#include` → `#содержит`;
`int main(...)` → `целое32 точка_входа(...)` с переименованием параметров в
`количество_аргументов`/`аргументы`; `i++`/`i +=` → `i = i + 1`/`i = i + …`.
Понижения: `do-while` → `пока (истина) { … если (!(усл)) { прервать } }`;
тернарник в присваивании/`return` → `если/иначе`; `switch` → `выбор`.

Отображение типов: `int`→`целое32`, `char`→`символ`, `unsigned int`→
`неподписанный целое32`, `int8_t…int64_t`→`целое8…целое64`, `uint8_t`→`байт`,
`float`/`double`→`вещественное`/`вещественное64`, `_Bool`→`логический`,
`void`→`ничего`, указатели сохраняются.

### Безопасная память: `malloc`/`free` → `срез`/`выделить`

Ручное управление памятью автоматически отображается на безопасную модель Konda,
чтобы `free()` исчез, а память стала безопасной (autofree транспилятора, проверка
границ, контроль владения). В пределах функции конвертер распознаёт выделение
кучи и переписывает его, если **все** использования указателя срез-совместимы:

| C | Konda |
|---|---|
| `int *a = malloc(N * sizeof(int))` | `срез<целое32> a = выделить(N)` |
| `int *p = malloc(sizeof(int))` | `срез<целое32> p = выделить(1)` |
| `int *a = calloc(N, sizeof(int))` | `срез<целое32> a = выделить(N)` |
| `*p` | `p[0]` |
| `p->поле` | `p[0].поле` |
| `free(p)` | *(выбрасывается — `срез` освобождается автоматически)* |

Убирать `free()` безопасно: autofree транспилятора освобождает `срез` ровно один
раз в конце области видимости, а границы `p[i]` теперь проверяются. Двойной
`free`, `free` в одной ветви и «use-after-free» из исходного C становятся
безопасными.

Выделение оставляется **сырым** (дословный `malloc`/`free`) с **предупреждением**,
если безопасно отобразить нельзя: указатель уходит из функции (`return p`,
передача в вызов, `p` в структуру), арифметика указателей, повторное присваивание
`p = …`, `realloc`, элемент — структура или указатель (`срез<T>` держит только
примитивы), либо размер не распознан как `sizeof`. Такой сырой код — по-прежнему
корректный C, но вне гарантий Konda; о нём предупреждает конвертер, а небезопасные
паттерны затем отвергает транспилятор.

### Ограничения

`goto`/метки, битовые поля, `#define`/условная компиляция, анонимные `struct`/
`union` в объявлении, определения вариативных функций, designated-инициализаторы,
`switch` с проваливанием (fallthrough), тернарник вне присваивания/`return`,
`++`/`--` как значение внутри выражения — не поддержаны (ошибка с позицией в
`.c`). `void*` приближается к `символ*`, `size_t` — к `неподписанный целое64`,
`const` отбрасывается; позиционная инициализация структур `{1, 2}` переводится
как есть и может быть отвергнута транспилятором (Konda ждёт `{ поле = … }`).

## Тесты

```sh
make test
```

Тесты проверяют:

- сборку транспилятора;
- что `тест.конда` (песочница) транспилируется и компилируется;
- передачу аргументов `printf` как есть, без подстановки `"%s\n"`;
- поддержку условия `>=`;
- приоритет операторов (`2 + 3 * 4 == 14`);
- логические `и`/`или` (→ `&&`/`||`): работу в условиях, приоритет (`и` крепче
  `или`), сохранение `и` как имени переменной, C-запись `&&`/`||`, многострочное
  условие в скобках;
- **побитовые операции** `& | ^ << >> ~` — только над целочисленными типами
  (float/указатель/срез отвергаются), нулевая рантайм-цена; приоритеты как в C
  (`| < ^ < & < сравнение < сдвиг < +/- < */%`); сдвиг на константу вне
  разрядности — ошибка компиляции, неконстантный — одна рантайм-проверка в отладке;
- вложенные `если` / `иначе`;
- объявления переменных и арифметику над ними;
- целочисленные типы по разрядности (`целое8/16/32/64` → `int8_t`…`int64_t`);
- остальные примитивы (`байт` → `uint8_t`, `вещественное` → `float`,
  `вещественное64` → `double`, `символ` → `char`, `логический` → `bool` c
  литералами `истина`/`ложь` → `true`/`false`);
- многоаргументные вызовы (`printf` с тремя и более аргументами);
- циклы `для`/`пока`, `прервать`/`продолжить` (→ `break`/`continue`, с проверкой
  «вне цикла»), присваивание, унарный минус и `вернуть`;
- указатели в теле функции (`символ* …` → `char *…`);
- модификатор `неподписанный` (`целоеN` → `uintN_t`) и ошибку для нецелого типа;
- ошибки таблицы символов: необъявленная переменная, повторное объявление,
  выход переменной за область видимости, индексирование не-указателя;
- ключевое слово `нуль` → `nullptr` в сгенерированном C;
- нулевое разыменование указателя на этапе компиляции: ошибку при
  индексировании указателя, доказанно равного `нуль` (явный `= нуль` и
  неинициализированный), зануление сырого указателя в `nullptr` в кодогене,
  отсутствие ложных срабатываний после переприсваивания валидным значением
  и для ненулевого указателя;
- путь-чувствительный анализ нулевости и инициализации: ошибку для указателя,
  присвоенного валидным значением лишь в одной ветви `если` или только в теле
  цикла (после блока может быть нулём), компиляцию при присваивании в обеих
  ветвях, такую же проверку для definite-assignment скаляров, и корректную
  работу sink-ветви (безусловный `вернуть` не вливается в слияние);
- снятие консервативности для доказуемо-выполняющихся циклов: `для` с
  константной границей и `для_каждого` по массиву засчитывают инициализацию в
  теле после цикла, тогда как динамическая или пустая граница — нет;
- инициализацию до использования: ошибку при чтении неинициализированного
  скаляра (напрямую и в выражении), отсутствие ложных срабатываний после
  присваивания и для занулённого массива;
- проверку диапазона константы-инициализатора: ошибки для константы и
  свёрнутого выражения вне диапазона типа, отрицательной константы в
  беззнаковый тип, и компиляцию граничных значений (255, -128, 65);
- ошибку для небезопасного доступа к `аргументы[1]`;
- инициализацию массива списком значений (полный/частичный литерал,
  выражения как элементы, ошибки переполнения и несовместимого типа,
  сохранение поведения зануления без инициализатора);
- авто-конверсию массива в срез при передаче в функцию: построение
  дескриптора без аллокаций, ошибку для несовместимого типа элемента,
  сохранение прежнего поведения для указательного и срез-аргумента;
- multi-file: вызов функции из соседнего файла (forward-прототип в C),
  структура из одного файла, используемая в объявлении переменной в другом
  (signature-pass по именам структур), независимость от порядка аргументов
  в командной строке;
- `для_каждого` по срезу и массиву: обход всех элементов, использование
  элемента в выражении (сумма), отсутствие рантайм-проверки границ в теле
  цикла, ошибка для не-коллекции, запрет move-из-внешней-области внутри
  тела;
- path-sensitive move владения: симметричный move в обеих ветвях
  `если`/`иначе`, ошибку для асимметричного move, sink-`вернуть` в ветви
  как корректный паттерн, запрет move из объемлющей области в теле цикла
  (для `пока` и `для`), разрешение move среза, объявленного внутри тела
  цикла, диагностику use-after-move с указанием строки места move;
- функцию-потребитель (`владеет`-параметр): запрет передачи владельца из
  объемлющей области в потребитель внутри тела цикла (`для` и `пока`) как
  use-after-move, и разрешение для среза, объявленного внутри тела цикла;
- транзитивную деструкцию структур: вызов деструктора поля-структуры при
  выходе из области (даже без собственного `[удалить]` у объемлющей
  структуры), обратный порядок освобождения нескольких полей и запрет
  копирования структуры, транзитивно содержащей деструктор. Глубина цепочки
  полей ограничена только числом объявленных структур в программе (не
  фиксированной константой): поле-структура по значению может ссылаться
  лишь на уже объявленную структуру, значит цикл структурно невозможен;
- `срез<T>` как поле структуры: структура становится владельцем памяти поля
  (инициализация только `выделить(N)`/`клон(x)` в литерале полей —
  `Тип о = { поле = выделить(N) }`), autofree освобождает поле при выходе
  структуры из области (наравне с деструктором и вложенными полями-структурами).
  Копирование/присваивание, передача по значению и возврат по значению такой
  структуры запрещены (двойной `free`) — те же правила, что для структуры с
  `[удалить]`. Поле можно заимствовать целиком (`ф(о.поле)`, `для_каждого (э
  из о.поле)`), но нельзя «увести» в отдельную переменную (`срез<T> с = о.поле`
  запрещено — структура остаётся единственным владельцем);
- перечисления (enum): базовое объявление и сравнение (авто-нумерация), явные
  значения с продолжением авто-инкремента, перечисление как параметр функции и
  поле структуры, `enum → целое` (расширение), ошибки для `целое → enum`,
  смешения разных перечислений и чтения перечисления до инициализации;
- возврат перечисления или структуры по значению (`Тип имя() { … вернуть х }`,
  `Тип рез = вызов()`): перечисление — свободно (обычный скаляр, деструкции
  не требует). Структуру можно возвращать по значению, если ей **не нужна**
  деструкция (ни своя, ни транзитивная через поле/срез-поле) — обычное
  копирование, как в C (в т.ч. когда в теле функции есть локальный владеющий
  срез: он освобождается перед возвратом, а возвращаемое значение сохраняется
  во временную корректного типа). Структуру, которой нужна деструкция,
  возвращать по значению запрещено явной ошибкой компиляции (была бы
  move-семантика возврата вроде `владеет`-параметра, но для неё нужен отдельный,
  более полный анализ; вместо тихой генерации небезопасного кода — понятный
  отказ);
- копирование структуры **без владения** по значению: присваивание `б = а`,
  инициализация при объявлении `Тип б = а`, инициализация поля литерала
  `{ поле = друг_структура }`, передача в функцию и возврат — всё это обычное
  побайтовое копирование (как в C). Для структуры, которой нужна деструкция
  (владеет срезом или имеет `[удалить]`, в т.ч. транзитивно), **любое**
  копирование значения запрещено явной ошибкой — во всех перечисленных формах,
  — потому что копия дескриптора владения привела бы к двойному освобождению;
- статические UB-проверки операторов: сдвиг вне разрядности, переполнение
  константного выражения, проверку свёрнутого сдвигом индекса на границы;
- защиту `выделить`: статическое переполнение размера и наличие проверки
  неудачи аллокации в прологе;
- запрет неявного сужения и явный каст: ошибку сужения непостоянного значения,
  компиляцию через `как<целое8>(…)` с эмиссией C-каста, свободное расширение;
- `выбор`/match по перечислению: компиляцию исчерпывающего выбора в `switch`,
  ошибку для пропущенного варианта (с именем), ветку `_`, ошибку для чужого
  варианта и путь-чувствительную инициализацию во всех ветвях;
- f-строки: интерполяцию переменной и выражения, спецификаторы по типу,
  экранирование литерального `%`, вложенные скобки, ошибку интерполяции среза и
  сохранение printf-семантики у обычных строк;
- безопасность приведений и интерполяции: запрет `целое → указатель` (анти-
  пример с фабрикацией адреса), разрешение `нуль`/указатель→целое/указатель→
  указатель, поимку индекса-через-приведение рантайм-проверкой среза (SIGABRT)
  и безопасное усечение длинной f-строки без переполнения буфера;
- безопасную обёртку над `аргументы`: динамический индекс вне границ прерывает
  программу (SIGABRT с сообщением), индекс в границах работает, а доказанный
  условием литеральный индекс остаётся прямым доступом без рантайм-проверки;
- запись в строковый литерал: ошибку прямой записи и через копию указателя,
  и разрешение записи в изменяемую строку `аргументы`;
- запрет decay массива в сырой указательный параметр (с подсказкой `срез<T>`) и
  безопасную замену тем же кодом через срез-параметр;
- рантайм-guard деления/сдвига: прерывание при делении на ноль и сдвиге вне
  разрядности в рантайме (с сообщением), работу ненулевого делителя и отсутствие
  guard-а при делителе-константе;
- рантайм-guard знакового переполнения `+/-/*`: прерывание при переполнении
  типа результата (с сообщением), работу без переполнения, отсутствие guard-а при
  константной арифметике (проверено статически) и при беззнаковых (перенос
  определён в C); типизацию суммы по старшему операнду (`целое8 + целое64 →
  целое64`, сужение в целое32 ловится);
- корректную ошибку для несуществующего файла.

## Что уже поддерживается

- директива `#содержит <...>`;
- **условная компиляция** `#if`/`#ifdef`/`#ifndef`/`#elif`/`#else`/`#endif` +
  `#define` (и русские `#если`/`#опр`/`#не_опр`/`#иначе_если`/`#иначе`/`#конец`/
  `#определить`), флаги `-D ИМЯ[=знач]` — безопасно (выбор веток, не подстановка;
  функциональных макросов нет — для констант `конст`). См. раздел выше;
- **индексация произвольного lvalue** для срезов: `структура.срез_поле[i]`, цепочки
  `а.б[i].в[j]` — чтение/запись «на месте» с проверкой границ (см. раздел);
- функция `точка_входа`;
- типы `целое8`, `целое16`, `целое32`, `целое64`, `байт`, `вещественное`
  (→ `float`), `вещественное64` (→ `double`), `символ`, `логический` (→ `bool`);
- `логический` с литералами `истина`/`ложь` (→ C23 `true`/`false`): годен как
  тип возврата, переменной, параметра и в условиях; в смешанной арифметике
  `вещественное64` доминирует над `вещественное`, оба — над целыми (как в C);
- `ничего` (→ `void`) — **только** как тип возврата функции, которая ничего не
  возвращает (`ничего печать(целое32 н) { … }`); в такой функции `вернуть` идёт
  без значения (или отсутствует). Для переменных/параметров `ничего` недопустимо;
- любая non-`ничего` функция (включая `точка_входа`) обязана возвращать значение
  **по всем путям** — иначе ошибка семантики (молчаливый fall-through в C — UB);
  анализ учитывает `если/иначе`, `выбор` и расходящийся `пока (истина)` без
  `прервать`;
- операторы управления циклом `прервать` (→ `break`) и `продолжить` (→ `continue`)
  с диагностикой использования вне цикла;
- модификатор `неподписанный` для целых типов (`неподписанный целое32` → `uint32_t`);
- завершение операторов переводом строки — `;` в конце строки не нужна
  (модель как в Go/Swift): перевод строки завершает оператор, если строка
  синтаксически закончена; строка, кончающаяся бинарным оператором или
  запятой, и любые переносы внутри `(…)`/`[…]` считаются продолжением.
  `;` остаётся как разделитель нескольких операторов в одной строке и как
  разделитель частей заголовка `для(…; …; …)`;
- `если` / `иначе`;
- логические операторы `и` (→ `&&`) и `или` (→ `||`) в условиях:
  `если (а > 0 и б > 0)`, `если (готов или таймаут)`. Приоритет как в C — `и`
  связывает крепче `или`, оба ниже сравнения. `или` — ключевое слово, а `и` —
  **контекстное**: логическим оператором оно является только между операндами,
  поэтому `и` по-прежнему можно использовать как имя переменной-счётчика
  (`для (целое32 и = 0; и < н; и = и + 1)`). C-запись `&&`/`||` тоже принимается.
  Многострочное условие в скобках переносится свободно (перевод строки внутри
  `(…)` — продолжение);
- циклы `пока` (→ `while`) и `для` (→ `for`);
- foreach: `для_каждого (имя из коллекция) { … }` — обход среза или массива
  фикс. размера; имя — фрешая переменная типа элемента (копия по значению,
  модификация не отражается на коллекции). Кодоген использует собственный
  индекс цикла, доступ прямой — без рантайм-проверок границ (нулевая
  стоимость, как у канонического `для` с BCE). Move-из-внешней-области
  внутри тела `для_каждого` запрещён по тем же причинам, что в обычном
  цикле;
- `вернуть` (→ `return`) — с выражением или без. Любая функция с непустым типом
  возврата (не `ничего`) обязана вернуть значение **по всем путям** — иначе
  ошибка семантики с позицией в `.конда` (fall-through в non-void функции C —
  это UB, возврат мусора; ловим статически, а не делегируем gcc). Анализ
  доказывает возврат рекурсивно: `вернуть`; `если`/`иначе`, где возвращают обе
  ветви; исчерпывающий `выбор`, где возвращают все ветви; бесконечный `пока (1)`.
  Правило единое — `точка_входа` тоже обязана завершиться `вернуть 0` (в отличие
  от C, где `main` может неявно вернуть 0);
- `ничего`-функция (void) от этого правила освобождена: возврата не требует;
- **безопасный указательный тип возврата** `Тип*` (borrow): функция может вернуть
  указатель-параметр, `аргументы`, строковый литерал или `нуль` — то есть память,
  которой владеет вызывающий/статика и которая переживёт возврат. Возврат
  указателя на локальные данные или результата C-функции запрещён статически
  (нет висячих указателей, нет ручного `free` у вызывающего). Пример:
  `символ* подлиннее(символ* а, символ* б) { … вернуть а }`;
- **обобщённые функции** (generics) через мономорфизацию: `T максимум<T>(T а, T б)`
  с вызовом `максимум<целое32>(3, 9)`. Пред-проход между разбором и семантикой
  генерирует по одной специализации на каждый уникальный тип-аргумент
  (`максимум_целое32`, `максимум_вещественное64`), подставляя конкретный тип и
  снимая `<T>`; вызовы переписываются на имя специализации. Как в Rust/C++ →
  никакого рантайма. Поддержаны несколько тип-параметров (`имя<T, U>`),
  тип-аргументы примитивов, структур, перечислений и указателей, транзитивные
  вызовы (обобщённая зовёт обобщённую) и рекурсия. Ошибки уровня Konda: неверное
  число тип-аргументов, неизвестный шаблон. Специализация статически проверяется
  семантикой как обычная функция (напр. `а > б` требует сравнимого типа);
- `статический` (→ `static`): для функций — внутренняя связь
  (`статический целое32 f()` → `static int32_t f()`); для локальных
  переменных — статическая длительность, значение сохраняется между вызовами
  (`статический целое32 счётчик = 0`). Инициализатор статической переменной
  должен быть константой; `точка_входа` и срезы статическими быть не могут;
- объявления простых переменных и присваивание существующим;
- указатели в объявлениях тела функции (`тип* имя`, `тип** имя`);
- вызовы функций;
- вызовы `printf(...)` (без особой обработки: формат-строку задаёт сам код);
- выражения с идентификаторами, числами, строками, унарным минусом и простыми операторами;
- доступ к массиву с числовым литералом, например `аргументы[1]`;
- таблица символов с областями видимости: проверка обращения к необъявленным
  переменным, повторного объявления в одной области и видимости по блокам;
- проверка совместимости типов по правилам C (без неявных приведений):
  ошибка при `указатель ← целое`, `целое ← указатель`, несовместимых типах
  указателей, несовместимом типе возврата, недопустимых операндах бинарного
  оператора и унарного `-`; смешение целых типов разрешено. **Арифметика
  указателей** (`ptr ± целое`, `ptr − ptr`) — только в `небезопасно` (см. раздел
  о гейтах выше);
- **ненулевые указатели по умолчанию**: обычному `T*` нельзя присвоить `нуль` и
  нельзя объявить без инициализатора; nullable — только `возможно<T*>`, которое
  разворачивается под охранником `если (п != нуль)` / `если (п == нуль) … иначе`
  (см. отдельный раздел). Оператор сравнения `!=` («не равно») поддержан;
- защита сырых указателей на этапе компиляции (путь-чувствительно): разыменование
  `возможно<T*>`, который может быть `нуль` хотя бы на одном пути исполнения, —
  ошибка. На стыке ветвей состояние сливается честно (после `если` ненулевой,
  только если сужен/присвоен на **всех** путях; sink-ветвь `вернуть` в слияние не
  вливается). Обычный `T*` ненулевой по построению — разыменуется без охранника;
  кодоген зануляет `возможно<T*>` без инициализатора в `nullptr`;
- инициализация до использования (definite assignment, путь-чувствительно):
  скалярную переменную нельзя читать, пока она не присвоена на **всех** путях.
  Присваивание лишь в одной ветви `если` (или только в теле цикла) не делает
  её инициализированной — чтение после блока ошибка «используется до
  инициализации». Закрывает чтение неопределённой памяти (UB) статически, без
  рантайма. Массивы/структуры зануляются и считаются инициализированными.
  Если цикл **доказуемо** выполняется хотя бы раз (`для` с константной/известной
  границей, истинной при входе, или `для_каждого` по массиву фикс. размера /
  срезу известной длины > 0), инициализация и ненулевость, установленные в
  теле, засчитываются и после цикла — консервативность снимается там, где это
  обоснованно;
- проверка диапазона константы-инициализатора: если сворачиваемая в константу
  величина не помещается в объявленный целый тип (`целое8 х = 1000`,
  `неподписанный целое8 = -1`, `целое16 = 300 * 200`), — ошибка «вне диапазона
  типа», вместо тихого усечения в C;
- статические проверки UB операторов (ноль рантайма): сдвиг на константное
  число бит вне разрядности типа (`1 << 40`), деление `МИН_типа / -1`,
  переполнение константного `+/-/*` относительно типа результата
  (`2000000000 + 2000000000`). Сдвиги/деления сворачиваются для проверки границ
  индексов;
- запрет неявного сужения целых: присваивание непостоянного значения в более
  узкий тип (`целое8 м = целое32_переменная`) — ошибка, нужно явное приведение
  `как<тип>(выражение)`. Расширение и константы (проверенные диапазоном)
  разрешены без приведения;
- защита `выделить`: переполнение размера `количество × sizeof(T)` для
  константного количества ловится статически; неудача аллокации в рантайме —
  детерминированный `abort` с сообщением (одна проверка в общем прологе, а не
  на каждый вызов);
- пользовательские структуры: `структура Имя { тип поле … }` → `typedef
  struct`; объявление `Имя п` (зануляется), доступ к полям `п.поле` (чтение и
  запись; для указателя на структуру `.` транслируется в `->`), вложенные
  структуры (`о.начало.х`), передача структуры в функцию по значению; проверка
  существования структуры, полей и типов полей;
- **союзы** (union): `союз Имя { тип поле … }` → `typedef union` (поля разделяют
  память). Объявление и литерал `{ поле = знач }` — как у структуры. Безопасность:
  **доступ к члену союза (`з.поле`) — только в `небезопасно`** (активный член не
  отслеживается, чтение неактивного = type confusion; безопасная альтернатива —
  `перечисление`+`выбор`); поле-срез в союзе запрещено (владение неоднозначно) и
  деструктор `[удалить=…]` на союзе запрещён;
- **указатели на функцию**: именованный тип `типфункции RET Имя(типы…)` →
  `typedef RET (*Имя)(типы…);`. Переменной/параметру такого типа присваивается
  имя функции с **совпадающей сигнатурой** (или `нуль`), вызов через указатель
  `ф(арг…)`, передача функции как аргумента (колбэки: `применить(сложить, …)`).
  Безопасность: вызов указателя, который может быть `нуль` — ошибка (крах в C);
  сигнатура присваиваемой функции сверяется с типом (возврат + типы параметров);
- **параметры по ссылке** `изменяемый` (C# `ref`) и `вывод` (C# `out`) — мутация
  переменной/структуры вызывающего на месте и «возврат нескольких значений»:
  `ничего разделить(целое32 а, целое32 б, вывод целое32 ч, вывод целое32 о)`,
  вызов `разделить(17, 5, ч, о)` — **без `&` и `*`** (кодоген сам передаёт `Тип*`,
  разыменовывает в теле, ставит `&` на вызове). Безопасность как в C#: аргумент
  обязан быть переменной (lvalue); `изменяемый` — инициализированной до вызова;
  `вывод` — **обязан быть присвоен по всем путям** до возврата (definite
  assignment, тот же путь-чувствительный анализ, что и «использование до
  инициализации»), после вызова переменная считается инициализированной. Ссылка
  не значение первого класса — её нельзя сохранить/вернуть/заалиасить, поэтому
  висячих указателей нет и borrow checker не нужен (в отличие от сырого `&`);
- **пространства имён**: `пространство Имя { … }` — объявления (функции, структуры,
  перечисления, союзы, типфункции) получают квалифицированные имена `Имя::член`,
  в C манглятся как `Имя__член` (плоский нейминг — как typedef'ы в C, нулевая
  рантайм-стоимость). Доступ снаружи — квалифицированный `Имя::член`; внутри
  пространства члены видны неквалифицированно; `использовать Имя` вносит имена
  пространства в область (можно писать `член` без префикса). Одноимённые типы в
  разных пространствах не конфликтуют (`А::Т` и `Б::Т` — разные C-имена);
- перечисления (enum): `перечисление Имя { ВАРИАНТ [= значение] … }` →
  `typedef enum { … } Имя;`. Варианты — по одному на строку (модель A), с
  необязательным явным целым значением (иначе предыдущее + 1, как в C).
  Переменная `Имя ц = ВАРИАНТ`, сравнение `ц == ВАРИАНТ`, перечисление как
  параметр функции и поле структуры. Типобезопасность: присваивание/сравнение
  только в пределах одного перечисления, `enum → целое` разрешено (расширение),
  `целое → enum` и смешение разных перечислений — ошибка; вариант — это
  целочисленная константа (сворачивается, годится для значений и сравнений).
  Переменную-перечисление, как любой скаляр, нужно инициализировать до чтения;
- `выбор`/match по перечислению с проверкой исчерпываемости: `выбор ц { ВАРИАНТ
  { … }  _ { … } }` → C `switch`. Семантика требует, чтобы были покрыты **все**
  варианты перечисления (или присутствовала ветка `_`), запрещает повторы и
  чужие варианты; путь-чувствительно сливает ветви (переменная, присвоенная во
  всех ветвях исчерпывающего выбора, считается инициализированной после него);
- **sum-типы (размеченные перечисления с данными)** — безопасная замена `союз`у:
  `перечисление Фигура { Круг(вещественное64 р)  Прямоугольник(вещественное64 ш,
  вещественное64 в)  Пусто }` (варианты на отдельных строках). Конструирование —
  как вызов: `Фигура ф = Круг(2.0)` (или голым именем `Пусто` без данных).
  Доступ к полезной нагрузке — **только через `выбор` с привязкой**, где активный
  вариант доказан на компиляции (модель Rust-enum): `выбор ф { Круг(р) { … р … }
  Прямоугольник(ш, в) { … } Пусто { … } }`. Кодоген — размеченный союз (`struct {
  тег; union { … } }`); тег нужен для `switch` (это семантика, не накладная) —
  **нулевая рантайм-цена**. Ограничения v1: поля варианта — примитивы/указатели
  (без `срез`: владение неоднозначно); опция-над-значениями (`возможно<целое>`)
  пока отдельно;
- явное приведение типа `как<тип>(выражение)` → C-каст `(тип)(…)`: для чисел,
  указателей и перечислений (escape-hatch для сужения и interop). Приведение
  **целого/перечисления к указателю запрещено** — иначе можно сфабриковать
  произвольный адрес и разыменовать его; для нулевого указателя есть `нуль`.
  Разрешены число↔число, указатель→целое (чтение адреса) и указатель→указатель
  **того же типа** (напр. смена знаковости). **Реинтерпретация указателя к
  другому типу** (`как<целое32*>(символ*)`, type confusion) — только в
  `небезопасно`;
- **иммутабельность `конст`**: `конст T имя = …` — переменную нельзя
  переприсвоить, изменить её элемент/поле или передать в `изменяемый`/`вывод`
  (см. отдельный раздел);
- **гейт небезопасных функций libc** (`strcpy`/`gets`/`sprintf`/…, неограниченная
  запись в буфер) — вызов только в `небезопасно`; безопасные (`printf`/`strlen`/…)
  не затронуты;
- интерполяция строк (f-строки, синтаксис как в Swift): `"Привет, \(имя).
  Через год тебе будет \(возраст + 1)."`. Внутри `\(…)` — любое выражение;
  спецификатор `printf` выбирается по его типу (символ→`%c`, вещественное→`%f`,
  `символ*`→`%s`, целые→`%d`/`%u`/`%lld`, перечисление→`%d`, указатель→`%p`),
  литеральный `%` экранируется. Значение f-строки имеет тип `символ*` и годится
  везде, где ждут строку. Кодоген — GNU statement-expression с `snprintf` в
  буфер, **поднятый в начало функции** (автоматическая память на стеке, а не
  `static`: потокобезопасно и реентерабельно; разные вставки не затирают друг
  друга). Быстрый путь: одиночная строковая вставка `"\(строка)"` эмитится как
  само выражение — без буфера и `snprintf`. Своя функция `printf` для
  пользователя не трогается; срез или структуру интерполировать нельзя. Обычные
  строки без `\(…)` передаются как есть (printf-семантика `%%` сохранена);
- литералы структур (designated initializers C): `Имя п = { поле = значение,
  … }` — порядок полей не важен, незаданные поля зануляются автоматически
  (как `{ .поле = знач, … }` в C). Семантика проверяет существование каждого
  поля и совместимость типов значений. Инициализировать литералом можно
  только саму структуру, не указатель на неё;
- деструкторы структур (RAII, в стиле Vala): атрибут `[удалить = функция]`
  перед `структура` задаёт деструктор, который компилятор вызывает
  автоматически (`удалить(&п)`) при выходе переменной из области и перед
  `вернуть`, в обратном порядке объявления — без рантайма. Структуры без
  деструктора не порождают вызовов. Деструкция **транзитивна**: если поле
  структуры само является структурой с деструктором (или с таким полем),
  деструктор поля вызывается автоматически (`удалить(&п.поле)`), даже если у
  объемлющей структуры собственного `[удалить]` нет; поля освобождаются в
  обратном порядке объявления. Структуру, которой нужна деструкция
  (собственная или транзитивно через поле), нельзя копировать/присваивать или
  передавать по значению — это привело бы к двойному вызову деструктора.
  Глубина цепочки полей-структур не ограничена искусственным числом: поле по
  значению может ссылаться только на уже объявленную структуру, поэтому цикл
  невозможен, и предел глубины — это число различных структур в программе (а
  не фиксированная константа, из-за которой раньше на вложенности глубже 16
  уровней деструктор молча пропускался — утечка без единой диагностики);
- срезы `срез<T>` — толстый указатель `{данные, длина}` с защитой от
  переполнения буфера: `выделить(N)` создаёт срез, `длина(с)` даёт длину,
  доступ `с[i]` проверяется. Константный выход за границу ловится на этапе
  компиляции (как `аргументы[N]`), динамический — в рантайме (одно сравнение
  на доступ, как в Go/Rust; при нарушении — сообщение и `abort`). Срезы
  передаются в функции по типу `срез<T>`;
- autofree срезов (RAII на этапе компиляции, без рантайма): владеющий срез
  (из `выделить`) освобождается автоматически (`free`) в конце своей области
  видимости и перед каждым `вернуть`, в обратном порядке объявления — утечек
  нет. Параметр-срез заимствуется и в функции не освобождается (владелец
  снаружи);
- перемещение владения срезом (move) — без рантайма и без аннотаций:
  `срез<T> б = а` переносит владение в `б` (копируется только дескриптор,
  данные не трогаются — бесплатно), источник `а` становится недействительным;
  использование после перемещения — ошибка компиляции (use-after-move) с
  указанием строки места move. Длина переносится вместе с владением, поэтому
  статическая проверка границ и элиминация работают и после move.
  Освобождается ровно один владелец. Передача в функцию остаётся
  заимствованием (срез после вызова жив). Перемещение заимствованного
  среза (параметра) запрещено;
- path-sensitive move в `если`/`иначе`: владение может уйти в обеих ветвях,
  тогда после if срез считается перемещённым (autofree не вставляет
  освобождение). Если move есть только в одной ветви, итог двусмыслен —
  ошибка компиляции с указанием конфликтующей ветви. Ветвь, заканчивающаяся
  безусловным `вернуть`, в слияние не участвует (sink) — паттерн
  «потребить + ранний return в одной ветви» корректен. Без рантайм-флагов:
  компилятор либо доказал согласованность, либо отказал;
- move источника из объемлющей области внутри тела цикла (`пока`/`для`)
  запрещён: на втором витке источник уже был бы перемещён → use-after-move.
  Запрет действует одинаково и для move-через-присваивание (`срез б = а`), и
  для move-через-передачу в `владеет`-параметр (`поглотить(а)`). Move среза,
  объявленного *внутри* тела цикла, по-прежнему разрешён — новый владелец
  создаётся на каждом витке;
- возврат среза из функции — передача владения вызывающему: функция с типом
  возврата `срез<T>` отдаёт владение наружу (внутри не освобождается), а
  принимающий `срез<T> д = создать(...)` получает владение и освобождает срез
  один раз. Вернуть заимствованный (параметр) срез нельзя — функция им не
  владеет;
- переприсваивание срезу: `а = выделить(N)` (переаллокация), `а = клон(б)`,
  `а = в` (move), `а = функция()` — прежние данные цели освобождаются, цель
  получает новое владение; ни утечки, ни double-free;
- `клон(а)` — явная глубокая копия среза: новый независимый владелец с
  собственными данными (источник остаётся жив). Нужен, только когда правда
  требуются две копии данных — move и заимствование остаются бесплатными;
- функция-потребитель: параметр `владеет срез<T>` забирает владение —
  функция освобождает срез сама, а передача в такой параметр перемещает
  аргумент (после вызова он недействителен). Обычный параметр-срез по-прежнему
  заимствует (срез после вызова жив);
- массивы фиксированного размера `тип имя[N]` (как `[T; N]` в Rust): размер
  известен на этапе компиляции, массив зануляется по умолчанию. Литеральный
  (или сворачиваемый в константу) индекс проверяется на этапе компиляции —
  выход за границу не скомпилируется; динамический индекс проверяется в
  рантайме (длина берётся через `sizeof`, в структуру не пишется — нулевая
  цена по памяти). При литеральных индексах рантайм-проверка не генерируется
  вовсе;
- инициализация массива списком значений: `тип имя[N] = { выр, выр, … }` →
  агрегатный инициализатор C `{ … }`. Если значений меньше размера,
  хвостовые элементы C-компилятор зануляет автоматически (без собственного
  рантайма). Семантика проверяет, что элементов не больше `N` и тип каждого
  совместим с типом массива;
- авто-конверсия массива в срез при передаче в функцию: если параметр —
  `срез<T>`, а аргумент — переменная-массив `T[N]`, на месте вызова кодоген
  строит дескриптор `(Срез){ .данные = а, .длина = N, .эл_размер = sizeof(T) }`.
  Массив остаётся на стеке — никаких аллокаций; одна функция-«вью»
  (`сумма`, `найти`, `печатать`) работает и с массивами, и со срезами.
  Семантика проверяет совпадение типа элемента — иначе ошибка компиляции;
- устранение лишних проверок границ (bounds-check elimination): если
  компилятор доказал, что индекс заведомо в границах, рантайм-проверка не
  генерируется. Покрыто: константный индекс в границах известной длины; и
  канонический цикл `для (и = 0; и < длина(с); и = и + ш)` либо
  `для (и = 0; и < N; и = и + ш)` (ш > 0), в теле которого индекс и срез не
  переназначаются, — все доступы `с[и]` внутри безопасны. В неоднозначных
  случаях проверка сохраняется (консервативно), поэтому защита не теряется;
- защита от деления на ноль на этапе компиляции: свёртка констант ловит
  `x / 0`, `x % 0`, делитель-переменную, константно равную нулю
  (`зк = 0; … / зк`), и нулевые константные выражения (`… / (а - б)` при
  `а == б`); без рантайм-проверок — нулевая стоимость, как в Rust. Значение
  отслеживается только у неизменяемых переменных (которым не присваивают
  заново), поэтому ложных срабатываний нет;
- строковые литералы только для чтения (статика): указатель, инициализированный
  строковым литералом (или скопированный из такого), нельзя использовать для
  записи `с[i] = …` — данные лежат в `.rodata`. Провенанс отслеживается; запись
  в изменяемую память (например строку из `аргументы`) разрешена;
- запрет «тихого» decay массива в сырой указательный параметр: передача массива
  фикс. размера в параметр `тип*` известной функции — ошибка (в теле терялась бы
  длина и была бы возможна запись за границы); нужно объявить параметр `срез<T>`
  (длина + рантайм-проверки). Для внешних C-функций (без сигнатуры) decay
  сохраняется;
- рантайм-guard арифметического UB для неконстантных операндов (одно сравнение,
  inline, без внешней библиотеки): деление/остаток на ноль и `МИН/−1`, сдвиг на
  число бит вне разрядности, а также **знаковое переполнение** `+`/`-`/`*`
  (checked-builtin, стиль Rust: результат проверяется на диапазон типа) — вместо
  `SIGFPE`/UB/тихого оборачивания даётся понятное сообщение и `abort`.
  Константные случаи по-прежнему ловятся статически, известные компилятору
  операнды не порождают проверку, а беззнаковое переполнение не проверяется
  (его перенос определён стандартом C). Все эти рантайм-guard'ы — в **отладочной**
  сборке; в `--релиз` они снимаются (арифметика как в C), статические же проверки
  остаются;
- безопасная обёртка над `аргументы` (argv): индексирование проверяется на
  границы относительно `количество_аргументов` (argc). Литеральный индекс,
  доказанный условием `количество_аргументов > N`/`>= N`, — прямой доступ без
  рантайма; недоказанный литерал — ошибка компиляции; динамический индекс —
  одна рантайм-проверка (выход за границы → понятное сообщение и `abort`,
  вместо неконтролируемого чтения argv). Проверка границ остаётся и в `--релиз`;
- диагностика ошибок с указанием строки и колонки (колонка считается
  по символам UTF-8, а не по байтам), а также ожидаемого и встреченного токена;
- защита самого транспилятора от переполнения стека на патологически сложном
  выражении (тысячи `+` подряд или тысячи вложенных скобок): разбор выражений
  рекурсивен, а результат — глубокое AST, которое затем несколько раз
  рекурсивно обходят типизация/кодоген; вместо segfault — понятная ошибка
  «выражение слишком сложное» с большим запасом по глубине относительно
  реального предела стека;
- CLI: `--version`/`-v` печатает версию; неизвестный `--флаг` отклоняется
  понятной ошибкой (не трактуется как имя файла); отсутствие `точка_входа`
  без `--библиотека` ловится до вызова `cc` понятной диагностикой (не сырой
  ошибкой линковщика `undefined reference to 'main'`);
- безопасный запуск компилятора C: `cc` вызывается через `execvp` с готовым
  argv (`fork`+`execvp`+`waitpid`), без промежуточной shell-команды. Раньше
  путь к `.c`/выходному файлу подставлялся в строку для `system()` — имя
  входного `.конда`-файла с кавычкой или `;`/`$(…)` вырывалось из кавычек и
  выполнялось как отдельная shell-команда (command injection). Теперь имя
  файла — всегда один элемент argv и не разбирается интерпретатором ни при
  каких символах в нём;
- **транспилятор видит функции из C-библиотек**: подключённые `#содержит`
  заголовки сканируются (эвристический разбор прототипов, с обходом вложенных
  `#include`), и извлечённые функции — `printf`, `strlen`, `malloc`, … —
  становятся «известными». Вызов функции, не найденной ни среди функций Konda,
  ни в заголовках, даёт **предупреждение уровня Konda с позицией в `.конда`**
  (строка/колонка исходника, а не сгенерированного `.c`) — до вызова `cc`.
  Предупреждение не прерывает сборку (escape-hatch для interop сохраняется:
  функцию, которую эвристика не нашла, всё ещё можно вызвать — её проверит
  `cc`). Для не-вариативных функций сверяется и число аргументов.

## Интеграция с IDE (LSP)

Библиотека **libkonda** предоставляет высокоуровневый API для редакторов —
анализ исходника **в памяти**, без записи файлов и без вызова `cc`
(заголовок [`libkonda_ide.h`](libkonda_ide.h)):

```c
#include "libkonda_ide.h"

КдРезультат *р = konda_анализ(исходник, длина, каталог_проекта);
// р->диагностики[] — ошибки и предупреждения с позицией (для подчёркиваний);
// р->функции[]    — функции Konda и внешние (C), с сигнатурой и позицией
//                   определения (для автодополнения, hover, goto-definition);
// р->типы[]       — структуры и перечисления с позицией (outline, goto-def).
konda_освободить_результат(р);
```

Этого достаточно для language server: подсветка ошибок, автодополнение (в т.ч.
библиотечных функций из подключённых заголовков), всплывающие подсказки с
сигнатурой и переход к определению. Всё через стабильный C-ABI, потребляемый
из любого языка по FFI.

## Потоки: безопасность через передачу владения

```konda
целое32 сумма(владеет срез<целое32> с)   // «владеет» — владение переходит потоку
{
    целое32 итог = 0
    для (целое32 и = 0; и < длина(с); и = и + 1) { итог = итог + с[и] }
    вернуть итог
}

срез<целое32> данные = выделить(4)
поток<целое32> п = запустить(сумма, данные)   // «данные» ПЕРЕМЕЩЕНЫ в поток
целое32 итог = присоединить(п)                // ждём и забираем результат
```

**Гонки данных невозможны по построению.** Поток получает данные только по
значению (копия) или перемещением владения — разделяемое изменяемое состояние
в этой модели просто **нельзя выразить**. Поэтому не нужны ни мьютексы, ни
атомики: безопасность статическая, рантайм — голый `pthread` (инвариант №1).
Проверено `-fsanitize=thread`: гонок нет.

Модель опирается на то, что в Konda уже есть: **нет сырого `&`** (нельзя
получить указатель на чужой кадр), **нет замыканий** (нечего неявно захватить)
и **есть move-анализ владения**. Shared-memory модель в стиле Rust потребовала
бы borrow-checker, который в Konda сознательно отсутствует.

Статические правила (все — ошибки компиляции, с позицией):

| правило | почему |
|---|---|
| функция потока не может иметь `изменяемый`/`вывод` | это указатель в кадр запускающего: поток пережил бы кадр → висячий указатель и гонка |
| срез-параметр обязан быть `владеет` | иначе запускающий сохранит доступ к той же памяти → гонка / use-after-free |
| указатель-параметр запрещён | заимствование чужой памяти с той же проблемой |
| использование среза после `запустить` | владение ушло потоку (ловит общий move-анализ) |
| повторное `присоединить` | дескриптор потреблён (use-after-move) |
| тип результата функции = `T` в `поток<T>` | иначе результат прочитался бы как чужой тип |

Реализация: `поток<T>` → C-тип `Поток { pthread_t ид; void *кадр; }`; на каждую
запущенную функцию кодоген печатает кадр `{ R рез; A арг0; }` (**результат
первым полем**), шим и запускатель. `присоединить` читает `*(T*)кадр`, зная `T`
из типа Konda, — поэтому один C-тип обслуживает любые `поток<T>` без обобщений.
`-pthread` добавляется к `cc` только когда потоки реально используются.

**Авто-join (RAII).** Дескриптор потока живёт по тем же правилам, что срез: не
присоединил явно — присоединится сам при выходе из области (иначе поток остался
бы отсоединённым, а кадр утёк). Явное `присоединить` снимает его с авто-join,
двойного join не будет.

```konda
поток<ничего> п = запустить(работник, 1)
// «присоединить» не вызвали — авто-join при выходе из области, утечки нет
```

## «параллельно для»: параллелизм данных

```konda
срез<целое32> с = выделить(1000000)
целое32 множитель = 10
параллельно для (целое32 и = 0; и < 1000000; и = и + 1) {
    с[и] = с[и] * множитель        // каждый поток пишет только «свои» элементы
}
```

Диапазон дробится на **непересекающиеся куски** (по одному на ядро), поэтому
потоки не делят ни одной ячейки — синхронизация не нужна вовсе, а гонок нет по
построению. Проверено TSan на 10⁶ элементов: 0 гонок. Общения между потоками
здесь нет в принципе — это и есть причина, по которой каналы не понадобились.

Замыканий в языке нет, поэтому кадр с захваченными переменными строит кодоген:
семантика собирает список внешних переменных, которые читает тело, кодоген
печатает `typedef struct { Срез с; целое32 множитель; long _нач; long _кон; }`
и шим **на уровне файла** (вложенная функция GNU потребовала бы трамплина на
исполняемом стеке — это SEGV на hardened-сборках).

Статические правила (ошибки компиляции):

| правило | почему |
|---|---|
| писать можно только в `база[и]` по переменной цикла | иначе непересечение кусков не доказать → гонка |
| запись во внешнюю переменную запрещена | потоки писали бы в одну ячейку → гонка |
| `прервать`/`продолжить`/`вернуть` запрещены | у кусков нет общего порядка |
| вызовы функций запрещены (кроме `длина`) | функция могла бы писать в общие данные (v1: нет анализа эффектов) |
| требуется каноническая форма `и = 0; и < предел; и = и + ш` | только так «и» пробегает диапазон ровно по разу |

Ограничения v1: у потока — не более одного аргумента и `срез` как результат не
поддержан; в `параллельно для` — нет вызовов функций в теле. Каналы `канал<T>`
**сознательно отложены** (см. CLAUDE.md): они дают только общение во время жизни
потока, но тянут рантайм-библиотеку и вносят взаимоблокировки — класс багов,
который Konda статически предотвратить не может. Fork-join + `параллельно для`
покрывают большинство реального параллелизма без этой цены.

## Что пока не поддерживается

- неявное сужение целых запрещено и требует `как<тип>(…)`; расширение
  (`целое8 → целое32`) свободно. Результат смешанной арифметики типизируется по
  **старшему** (более широкому) операнду, как в C: `целое8 + целое64 → целое64`,
  поэтому `целое32 в = целое8 + целое64` ловится как сужение (нужен
  `как<целое32>(…)`) — тихого усечения суммы больше нет;
- защита формата `printf` от неконстантной строки (format-string): первый
  аргумент сейчас передаётся как есть по дизайну (`printf(аргументы[1])`);
  ограничение его строковым литералом — отдельный строгий режим;
- осознанно небезопасные операции доступны **только внутри `небезопасно { }`**
  (в безопасном коде — статическая ошибка, без рантайм-цены): индексирование и
  арифметика **сырого** указателя `тип*`, реинтерпретация указателя к другому типу
  (`как<целое64*>(символ*)`, type confusion), доступ к члену `союз`а, вызов
  небезопасных функций libc (`strcpy`/`gets`/…). Безопасный путь — `срез<T>`,
  массивы фикс. размера и `аргументы` (все с проверкой границ), а также
  `перечисление`+`выбор` вместо `союз`. Остаётся по дизайну (ты просил не трогать
  `printf` до своей stdlib): `printf`-формат передаётся как есть (`%s` на
  не-`\0`-строке — over-read). Уже закрыто статически: указатели ненулевые по
  умолчанию (nullable — только `возможно<T*>` с разворотом под охранником),
  создание указателя из целого **запрещено**, decay массива в сырой указательный
  параметр **запрещён**, запись в строковый литерал **запрещена**, деление/сдвиг
  с UB — рантайм-guard (в релизе снят, но `-fwrapv` делает переполнение
  определённым, не UB), знаковое переполнение `+`/`-`/`*` — рантайм-guard в
  отладке (стиль Rust: abort; беззнаковый перенос определён в C);
  срезы/массивы/`аргументы` проверяют границы даже при индексе через приведение;
- пользовательские функции с вызовом по кириллическому имени (транслитерация);
- перемещение среза из объемлющей области **внутри цикла** запрещено: на
  втором витке источник уже был бы перемещён на первом (use-after-move), а
  однопроходный анализ не проверяет это как fix-point по итерациям — сознательное
  ограничение анализа, а не пробел. Вне цикла (в `если`/`иначе`, при вызове
  функции, в объявлении) move из объемлющей области работает свободно;
- массивы переменной длины и многомерные; вывод размера из литерала
  (`целое32 а[] = {1,2,3}`);
- автоматическая конвертация массива в срез действует только для параметра
  функции; в локальной переменной (`срез<T> с = а`) пока не поддерживается;
- слияние владения при сложном control-flow за пределами «sink-вернуть в
  ветви»: вложенный `если`, оба внутренних блока которого возвращают,
  считается sink только если последний оператор внешней ветви — сам
  `вернуть`; более глубокий анализ путей пока не делается (сама вложенная
  конструкция при этом безопасна: код после неё физически недостижим в
  сгенерированном C, поэтому анализ просто не доказывает то, что и так не
  выполнится, а не пропускает реальный use-after-move);
- индексация произвольного lvalue **реализована для срезов**: `структура.срез_поле[i]`,
  цепочки `а.б[i].в[j]` — чтение и запись «на месте» с проверкой границ (длина
  берётся из самого среза). Ограничение: базой не может быть результат вызова
  (`функция()[i]`) — индексировался бы владеющий временный срез, который никто не
  освободит (утечка); присвойте результат в переменную. Массив/указатель «на
  месте» тоже не индексируются (нет длины/символа) — присвойте в переменную;
- пространства имён v1: **один уровень** (без вложенности); обращение к члену
  должно быть после его объявления (declaration-before-use, как для типов);
  обобщённые функции **внутри** пространства пока не поддержаны (пред-проход
  тип-параметров сканирует по глубине скобок = 0); при импорте двух пространств с
  одинаковым именем побеждает первое `использовать` (уточняйте квалификацией);
- обобщённые функции v1: тип-аргументы **явные** (вывод из аргументов не
  делается — `максимум<целое32>(…)`, не `максимум(…)`); обобщение работает в
  пределах **одного файла** (шаблон и вызовы в разных файлах пока не
  связываются: пред-проход тип-параметров идёт по каждому файлу отдельно);
  обобщённых **структур** (`структура Пара<T>`) и `срез<T>` в позиции
  тип-аргумента пока нет;
- указательный тип возврата функции **реализован безопасно** (borrow): `Тип* ф(…)`
  можно вернуть только указатель на память, которая переживёт возврат и
  принадлежит вызывающему/статике — параметр-указатель, `аргументы`, строковый
  литерал, `нуль` (или их индексация). Возврат указателя на локальные данные,
  результата C-функции или неизвестного происхождения — ошибка семантики (иначе
  висячий указатель / ручное управление памятью). Провенанс отслеживается
  по присваиваниям и сливается по путям (`если/иначе`, цикл) консервативно:
  безопасен, только если безопасен на всех путях. Возврат — заимствование:
  вызывающий не получает обязанности освобождать. Ограничение v1: возврат
  указателя, вычисленного арифметикой, и передача указателя на локаль наружу
  иначе как borrow'ом не поддержаны (взятия адреса `&` в языке нет — см. ниже);
- оператор взятия адреса `&x`: получить указатель на локальную переменную
  нельзя, поэтому параметр-указатель `Тип*` пользовательской функции на
  практике заполнить нечем (указатели приходят от `аргументы`, строковых
  литералов и C-функций). Структуры поэтому передаются и возвращаются **по
  значению** (см. выше); указательные параметры сейчас нужны в основном для
  авто-генерируемых компилятором вызовов деструкторов;
- **семантические** ошибки собираются батчем: анализ не останавливается на
  первой, а восстанавливается на границе оператора и функции (снимок/восстановление
  scratch-состояния, сброс области видимости) и продолжает проверку. И CLI
  (`Всего ошибок: N`), и `konda_анализ` для IDE отдают весь список сразу —
  подсветить все ошибки разом. Не фатальные **предупреждения** (например, вызов
  необъявленной функции) собираются так же. Ограничение: восстановление возможно
  цепным — одна настоящая ошибка иногда порождает пару производных; и **разбор**
  (парсер) по-прежнему останавливается на первой ошибке (одна `Диагностика`), так
  что синтаксические ошибки показываются по одной;
- сканер C-заголовков — эвристический (разбор прототипов текстом, не полный
  препроцессор C): функции за макросами (например скрытые `#define`-обёрткой)
  или в нестандартных каталогах включения могут не найтись — тогда вызов такой
  функции даст предупреждение, хотя код валиден (сборку это не ломает — `cc`
  скомпилирует). Системные пути — `/usr/include`, `/usr/local/include`; типы
  аргументов у C-функций не сверяются (только количество у не-вариативных);
- стабильная генерация ASCII-имен для всех кириллических идентификаторов.

## Архитектура

Транспиляция идёт конвейером из трёх проходов:

```
.конда (один или несколько) → лексер → разбор.c (AST на файл)
   → слияние AST → семантика.c (границы, типы) → кодоген.c → .c → cc
```

При нескольких входных файлах разбор делается независимо для каждого, потом
top-level дети сливаются в один общий `УЗЕЛ_ПРОГРАММА`. Перед разбором
делается пред-сканирование токенов всех файлов на «структура Имя», чтобы
парсер каждого файла знал о пользовательских типах из соседних. Семантика
и кодоген дальше работают с одним общим AST — никаких заголовков и
extern-деклараций между .конда не нужно.

Структура узла AST — в `аст.h`/`аст.c` поверх универсального `ДинМассив`.

Path-sensitive анализ владения (snapshot/merge для срезов в `если`/`иначе`,
sink-детектор для `вернуть`) вынесен в отдельный модуль `владение.c` с
внутренним заголовком `владение.h`. Семантика держит `Анализатор`/`Символ`
приватными и предоставляет модулю владения 5 узких аксессоров — изменение
анализа путей не требует править остальные проходы.

Весь конвейер собран в разделяемую библиотеку **libkonda** (`Собранное/libkonda.so`)
с единым публичным заголовком `konda.h`. `основа.c` — это CLI поверх библиотеки;
любая программа может подключить `konda.h` и вызвать конвейер напрямую.

## Ближайший план

Базовая безопасность памяти реализована: autofree (RAII в кодогене), move/borrow
с защитой от use-after-move/double-free, проверка границ срезов/массивов/argv,
ненулевые указатели по умолчанию + `возможно<T*>`, гейт всех сырых операций за
`небезопасно`, иммутабельность `конст`. Дальнейшие приоритеты — полнота и удобство
языка (безопасность уже в хорошем состоянии):

1. ~~**Sum-типы с данными**~~ — ✅ реализованы: размеченные `перечисление Имя {
   Вариант(тип…) … }` с привязкой payload в ветках `выбор` (см. раздел выше).
   Дальше на их основе — `Результат<T,E>` и опция-над-значениями (`возможно<значение>`).
2. ~~**Индексация произвольного lvalue**~~ — ✅ реализована для срезов
   (`структура.поле[i]`, цепочки); `функция()[i]` осознанно запрещён (утечка
   владеющего временного).
3. **Стабильная транслитерация кириллических имён в ASCII** в кодогене (сейчас
   идентификаторы уходят в C как UTF-8 — работает, но зависит от `cc`/локали).
4. Полнота: обобщённые структуры (`структура Пара<T>`), связывание обобщений между
   файлами, вывод размера массива из литерала (`целое32 а[] = {1,2,3}`), локальное
   `срез<T> с = массив`, батч-ошибки парсера.

Правило неизменно: расширять тесты перед каждым изменением синтаксиса. Целевая
архитектура описана в `АРХИТЕКТУРА_ТРАНСПИЛЯТОРА.md`.
